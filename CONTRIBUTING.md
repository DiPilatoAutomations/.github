# Contributing

Thanks for wanting to make this better. House rules — they're short:

1. **Open an issue first** for anything bigger than a typo fix, so we agree on the approach before you write code.
2. **No placeholders, no mock data.** `TODO`, `YOUR_API_KEY`, fake testimonials, and hardcoded sample rows don't merge here.
3. **No silent fallbacks.** A failure path must retry, alert, or surface loudly — never return a quiet default.
4. **Prove it works.** PRs include what you ran and what you saw: command output, logs, screenshots. "It should work" isn't evidence.
5. **Match the codebase.** TypeScript by default, Zod at the boundaries, plain language in docs.

Every PR gets reviewed. Small, focused PRs merge fast; grab-bag PRs get sent back.
